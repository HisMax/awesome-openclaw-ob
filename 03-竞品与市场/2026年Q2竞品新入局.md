## AWS Kiro IDE

### 基本信息

| 维度 | 详情 |
|------|------|
| **发布日期** | 2026-05-07（国际发布） |
| **开发商** | Amazon / AWS |
| **定位** | Spec-driven Agentic IDE |
| **底层模型** | Claude Sonnet + Amazon Nova（via Bedrock） |
| **前身** | 取代 Amazon Q Developer |

### Spec-driven Development：核心差异化

Kiro 的核心理念是：**先写规范，再写代码**。这与其他所有 AI IDE 的"先写 prompt，AI 直接出代码"路线截然不同。

#### 三步工作流

```
1. requirements.md  ← EARS 标注法的用户故事 + 验收标准
       ↓
2. design.md        ← 系统架构、时序图、组件拆解
       ↓
3. tasks.md         ← 编号实施清单，Agent 逐项执行
```

在写任何一行代码之前，Kiro 的 Agent 会先生成这三个结构化文档。用户审核并调整后，Agent 才按照 tasks.md 的清单逐步实现。

#### 为什么这很重要

这直接回应了 [[Agentic Coding]] 的一个核心痛点：**Agent 写的代码在本地能跑，但偏离架构、遗漏边界条件、在生产环境崩溃**。

Spec-driven 方法通过在编码前锁定需求和架构，将这些问题前置到"规划阶段"而非"调试阶段"。代价是工作流更长，但对企业级项目来说，这个代价可能是值得的。

### Agent Hooks：事件驱动自动化

| 触发事件 | 自动动作 | 说明 |
|---------|---------|------|
| 保存 React 组件 | 测试文件自动更新 | 测试与代码同步 |
| 修改 API 端点 | README 自动刷新 | 文档与代码同步 |
| Pre-commit 触发 | 安全扫描自动运行 | 安全左移 |

Agent Hooks 的理念与 [[Cursor Automations]] 类似，但 Kiro 的实现更偏向"软件工程最佳实践的自动化"，而非"Agent 的持续后台运行"。

### 对 OpenClaw 的影响

Kiro 的 spec-driven 路线对 OpenClaw 既是威胁也是启发：
- **威胁**：如果企业客户发现 spec-driven 方法显著减少了 Agent 代码的质量问题，他们可能优先选择 Kiro 而非直接 prompt-to-code 的方案
- **启发**：OpenClaw 可以在自己的工作流中集成 spec 生成步骤，而不是完全依赖"先规划后执行"这种更重的流程

## Gemini CLI → Antigravity CLI

### 事件经过

这是 2026 年开源社区最具争议的事件之一。

| 时间 | 事件 |
|------|------|
| 2025-06 | Google 以 Apache 2.0 开源 Gemini CLI |
| 2025-06 ~ 2026-05 | 社区贡献 6,000+ PRs，突破 **100,000 GitHub Stars** |
| 2026-05-19 | Google I/O 2026 宣布 Antigravity CLI |
| 2026-05-20 | 正式公告：Gemini CLI 向 Antigravity CLI 过渡 |
| 2026-06-18 | Gemini CLI 停止为免费和个人用户提供服务 |

### Antigravity CLI 与 Gemini CLI 的对比

| 维度 | Gemini CLI | Antigravity CLI |
|------|-----------|----------------|
| 开源协议 | Apache 2.0 | **闭源** |
| 运行时 | Node.js | **Go（原生二进制）** |
| 命令 | `gemini` | `agy` |
| 免费配额 | ~1,000 请求/天 | **~20 请求/天（-98%）** |
| 多 Agent | 有限 | **原生多 Agent 支持** |
| 安装 | `npm install` | 单文件二进制 |

### 争议核心

**信任危机**：一年内从 Apache 2.0 开源到闭源替代，Google 破坏了开源社区最基本的信任契约。6,000+ 个 PR 的贡献者感觉自己的工作被"白嫖"。

**配额暴跌**：从每天 ~1,000 请求到 ~20 请求，降幅 98%。这使得免费用户基本无法使用 Antigravity CLI 做任何有意义的工作。

**企业例外**：Gemini Code Assist Standard / Enterprise 许可的组织用户不受影响，仍可通过 Gemini CLI 访问。这强化了"开源引流、企业收割"的叙事。

### 开源社区的回应

这个事件直接推动了 [[#OpenCode 生态崛起|OpenCode]] 的加速增长——开发者开始寻找"真正不会被闭源"的替代品。

对 OpenClaw 的启示：**开源承诺一旦建立，违反的代价是灾难性的**。如果 OpenClaw 有开源组件，维护开源承诺的一致性比短期商业利益更重要。

### Gemini CLI 受影响用户的去向

| 去向 | 说明 |
|------|------|
| Antigravity CLI（付费） | 接受新规则，转为付费用户 |
| [[Claude Code 2026年Q2更新\|Claude Code]] | Anthropic 生态 |
| [[#OpenCode 生态崛起\|OpenCode]] | 开源替代，增长最快 |
| 继续用 Gemini CLI（企业版） | 仅限 Code Assist 许可用户 |

Gemini 3.5 Flash 的定位很清晰：**不追求与 [[Claude Opus 4.7-4.8 发布|Opus 4.8]] 或 [[GPT-5.5 发布|GPT-5.5]] 正面竞争，而是在速度和性价比上建立差异化**。

Gemini 3.5 Flash 的存在为 OpenClaw 的多模型策略提供了一个明确的"高速低成本"选项——用 Opus 做精细任务、用 Gemini 3.5 Flash 做批量任务，可以在不牺牲太多质量的前提下大幅降低成本。

## OpenCode 生态崛起

### 基本信息

| 维度 | 详情 |
|------|------|
| **类型** | 开源 CLI AI 编码 Agent |
| **GitHub Stars** | **~165K**（最高星标的开源 AI 编码 Agent） |
| **月活用户** | **750 万+** |
| **增长速度** | 2026 年增长最快的开源开发工具 |
| **支持模型** | 75+ LLM 端点 |

### 支持的模型生态

| 提供商 | 说明 |
|--------|------|
| Anthropic | Claude 全系列 |
| OpenAI | GPT 全系列 |
| Google | Gemini 全系列 |
| AWS Bedrock | 通过 Bedrock 访问 |
| Azure OpenAI | 企业 Azure 端点 |
| OpenRouter | 聚合路由 |
| xAI (Grok) | — |
| Ollama（本地） | 本地模型运行 |
| 任何 OpenAI 兼容端点 | 广泛兼容 |

75+ 端点的支持是 OpenCode 的核心优势——**真正的模型无关性**。用户可以自由选择任何提供商，包括在本地运行开源模型。

### 增长驱动力

1. **Gemini CLI 闭源的受益者**：开发者从 Gemini CLI 迁移出来后，OpenCode 是最自然的选择——同样是 CLI、同样开源、但不会被闭源

2. **Claude Code 的开源替代**：Claude Code 绑定 Anthropic 生态，OpenCode 不绑定任何提供商，吸引了重视灵活性的开发者

3. **终端 Agent 趋势**：2026 年的一个明确趋势是"终端 Agent 胜出"——CLI 原生的编码 Agent 在自动化和可编排性上优于 IDE 嵌入式方案

4. **社区驱动**：作为开源项目，OpenCode 受益于社区贡献的插件、集成和改进

### OpenCode vs Claude Code

| 维度 | OpenCode | Claude Code |
|------|----------|-------------|
| 开源 | 是（完全开源） | 部分开源（SDK） |
| 模型锁定 | 无（75+ 端点） | Anthropic only |
| 月活 | 750 万+ | — |
| 多 Agent | 基础支持 | [[Claude Code 2026年Q2更新\|Dynamic Workflows 1,000 子 Agent]] |
| 定价 | 免费（自带模型 API key） | API 按量 / Max 固定月费 |
| 企业支持 | 社区驱动 | Anthropic 官方支持 |

OpenCode 在规模和灵活性上领先，Claude Code 在 Agent 编排深度和企业支持上领先。两者不完全是替代关系，更像是面向不同用户群的互补产品。

### 对 OpenClaw 的影响

OpenCode 的 165K stars 和 750 万月活证明了一件事：**开源 + 模型无关 + CLI 原生**的组合有巨大的市场需求。

OpenClaw 的启示：
1. **开源策略**：如果 OpenClaw 开源核心编排引擎，可能复制 OpenCode 的社区增长效应
2. **模型无关性是刚需**：750 万用户选择 OpenCode 的一个核心原因是不被锁定。OpenClaw 必须支持多模型
3. **CLI 优先**：IDE 集成可以后做，CLI 优先的策略能更快覆盖自动化和 CI/CD 场景

## 综合影响与 OpenClaw 机会

### 市场信号

1. **$10.8B 市场仍在快速增长**：42% YoY 意味着蛋糕还在变大，新进入者仍有空间
2. **三种范式正在竞争**：Agent-first（Devin）、Editor-first（Cursor）、Spec-first（Kiro），行业尚未收敛
3. **开源是有效的增长策略**：OpenCode 证明了开源 + 社区可以快速获取大规模用户
4. **信任是核心瓶颈**：84% 采用 / 29% 信任的鸿沟是最大的未解决问题

### OpenClaw 的战略定位建议

| 方向 | 理由 | 参照 |
|------|------|------|
| 多模型路由 | 避免供应商锁定是用户刚需 | OpenCode、Cursor |
| 开源核心组件 | 社区驱动增长最有效 | OpenCode |
| 信任优先 | 29% 信任率是行业最大痛点 | [[Anthropic Mythos 模型\|Opus 4.8 诚实度]] |
| CLI 优先 + IDE 集成 | 覆盖最广的用户场景 | Claude Code + Cursor |
| Spec-optional 工作流 | 不像 Kiro 强制 spec，但支持 spec 集成 | Kiro |
