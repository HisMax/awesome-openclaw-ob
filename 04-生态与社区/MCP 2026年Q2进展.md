# MCP 2026年Q2进展


![[assets/mcp-q2.jpg]]

> **一句话总结**：2026 年 Q2，MCP 发布了自诞生以来最大规模的协议修订——通过 6 个 SEP 实现协议层无状态化、引入 Extensions 框架、MCP Apps（HTML UI）、Tasks 扩展和 12 个月弃用生命周期，同时 MCP Registry 服务器数破 9,600+。MCP 正在从"实验性协议"升级为"生产级基础设施标准"。

## Release Candidate 概览（2026-07-28）

| 维度 | 详情 |
|------|------|
| **锁定日期** | 2026 年 5 月 21 日 |
| **RC 发布** | 2026 年 Q2 |
| **正式规范发布** | 2026 年 7 月 28 日 |
| **验证窗口** | 10 周（SDK 实现和验证） |
| **性质** | MCP 启动以来最大规模协议修订 |

这次 RC 交付了 [[MCP 2026 路线图]] 中承诺的核心能力：无状态核心 + Extensions 框架 + 授权加固 + 弃用策略。

## 核心变化：协议层无状态化

### 6 个 SEP 协同实现

无状态化是本次 RC 的头号变化。6 个 Specification Enhancement Proposals（SEP）协同工作，完成了 2025 年 12 月《The Future of MCP Transports》中提出的计划：

| SEP | 变化 | 影响 |
|-----|------|------|
| **SEP-2575** | 移除 `initialize`/`initialized` 握手；Client 信息和能力通过每个请求的 `_meta` 传递 | 消除启动阶段的状态依赖 |
| **SEP-2567** | 移除 `Mcp-Session-Id` 头和协议级 Session | 消除 sticky routing 和共享 Session Store 的需求 |
| **SEP-2260** | 服务器发起的请求仅在活跃的客户端请求处理期间发生 | 简化服务器端状态管理 |
| **SEP-2322** | 多轮请求用 `InputRequiredResult` + `requestState` 替代 SSE 流 | 无状态的多轮交互 |
| **SEP-2243** | 强制 `Mcp-Method` 和 `Mcp-Name` 头 | 负载均衡器无需解析请求体即可路由 |
| **SEP-414** | 标准化 W3C Trace Context（`traceparent`、`tracestate`、`baggage`）在 `_meta` 中传播 | 分布式追踪标准化 |

### 无状态化的实际意义

用一句话概括官方表述："一个之前需要 sticky sessions、共享 session store 和深度包检查的远程 MCP 服务器，现在可以运行在普通的 round-robin 负载均衡器后面。"

这意味着：
- **水平扩展**：任何 MCP 请求可以落到任何服务器实例上
- **部署简化**：不再需要 Redis/Memcached 等 Session 存储
- **成本降低**：可以使用最廉价的 HTTP 基础设施
- **容错提升**：单实例故障不会丢失 Session 状态

### 有状态应用怎么办？

协议无状态 ≠ 应用无状态。应用可以通过显式 handle（句柄）实现状态——模型在 tool call 之间传递标识符作为普通参数，让状态**可见**而非**隐藏在传输元数据中**。这是一个优雅的架构决策：状态的存在被提升到应用层可见，而非隐藏在协议层黑箱中。

## Extensions 框架（SEP-2133）

MCP 获得了正式的扩展系统，这是协议治理成熟的标志：

| 维度 | 详情 |
|------|------|
| **标识** | 反向 DNS ID（如 `io.modelcontextprotocol.ext.apps`） |
| **协商** | 通过 capability maps 在连接时协商 |
| **代码仓库** | 独立的 `ext-*` 仓库，委派维护者 |
| **版本** | 独立于核心规范版本 |
| **治理** | SEP 流程新增 Extensions Track |

Extensions 框架的设计理念类似于 Web 浏览器的扩展系统：核心协议保持精简稳定，新功能通过扩展独立演进。这解决了 [[MCP 协议]] 早期"什么都往核心塞"的问题。

## 官方扩展

### MCP Apps（SEP-1865）

MCP Apps 让服务器提供交互式 HTML 界面，在沙箱化的 iframe 中渲染：

- 工具预先声明 UI 模板，Host 可以预取、缓存、安全审查
- 渲染的 UI 通过 MCP 通用的 JSON-RPC 协议与 Host 通信
- 这是 [[MCP 协议 2026年3月进展|3 月 Cursor MCP Apps 落地]] 的协议层标准化

MCP Apps 的意义在于让 Agent 从"纯文本"进化为"可视化交互"——MCP 服务器本身可以成为一个应用。

### Tasks 扩展

Tasks 从实验性核心功能（2025-11-25 引入）升级为正式扩展：

| 变化 | 说明 |
|------|------|
| **触发方式** | 服务器响应 `tools/call` 时返回 task handle |
| **客户端控制** | `tasks/get`、`tasks/update`、`tasks/cancel` |
| **创建权** | 服务器决定何时将调用转为 Task |
| **移除** | `tasks/list` 被移除（无 Session 无法确定 scope） |

Tasks 扩展为**长时运行工作流**提供了标准化模式，与 [[Agent 长时运行与 ACP 演进|Agent 长时运行趋势]] 高度吻合。

### 弃用的核心功能

三个核心功能正式进入弃用通道（SEP-2577）：

| 弃用功能 | 替代方案 | 弃用原因 |
|----------|----------|----------|
| **Roots** | 工具参数、资源 URI 或配置 | 概念过于模糊 |
| **Sampling** | 直接调用 LLM Provider API | 与无状态化冲突 |
| **Logging** | stderr 或 OpenTelemetry | 标准化日志方案更成熟 |

弃用策略保证最少 12 个月的过渡期（SEP-2596），这是协议成熟度的重要标志。

## 授权加固（6 个 SEP）

| SEP | 变化 |
|-----|------|
| SEP-2468 | 客户端必须验证 `iss` 参数（RFC 9207），防止 mix-up 攻击 |
| SEP-837 | 客户端声明 OpenID Connect `application_type` |
| SEP-2352 | 凭证绑定到 issuer；资源迁移需重新注册 |
| SEP-2207 | 文档化 OpenID Connect 服务器的 refresh token 请求 |
| SEP-2350 | 明确 step-up authentication 中的 scope 累积行为 |
| SEP-2351 | 规范 `.well-known` discovery suffix 使用 |

授权加固与 [[MCP 2026 路线图]] 中"企业就绪"方向一致——企业级部署需要与现有 OAuth/OIDC 基础设施无缝集成。

## 其他技术变化

| SEP | 变化 |
|-----|------|
| SEP-2549 | List/Resource 响应新增 `ttlMs` 和 `cacheScope`（对标 HTTP `Cache-Control`） |

## 关键洞察

### 无状态化是 MCP 的"HTTP 时刻"

早期的 Web 协议也经历过有状态到无状态的演进。HTTP 的无状态设计是 Web 能够无限扩展的根本原因——MCP 做出了同样的选择。这一决策让 MCP 服务器可以像 HTTP API 一样部署在任何云基础设施上，极大降低了企业采纳门槛。

### Extensions 框架防止"协议膨胀"

将 Tasks、MCP Apps 等功能从核心规范移入扩展，说明 MCP 团队从 Web 标准的经验中学到了教训——核心规范越精简越好，复杂功能通过扩展独立演进。这与 HTTP 核心保持简单、通过 Header 和 Extension 扩展功能的思路一脉相承。

### 12 个月弃用策略是成熟度信号

一个协议愿意正式弃用自己的功能（而不是无限向后兼容），说明它有信心和治理能力来管理演进——这是从"草创协议"到"行业标准"的关键转变。
