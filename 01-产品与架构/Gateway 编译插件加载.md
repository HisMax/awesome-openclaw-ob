# Gateway 编译插件加载

## 一句话理解

> 以前 Gateway 启动时像"现场做菜"——每次都要从源码编译插件再加载；现在改成"预制菜加热"——从编译好的 `dist/extensions` 直接加载，WhatsApp 冷启动从"几十秒"缩短到"几秒"。

## 核心变更

v2026.3.22 将捆绑频道插件的加载方式从**运行时编译**改为**预编译分发**。所有捆绑的频道插件（WhatsApp、Telegram、Discord、Matrix 等）现在从 `dist/extensions` 目录加载已编译的 JavaScript 文件，而非启动时从 TypeScript 源码实时编译。

此变更由 **@ngutman** 在 PR #47560 中贡献。

### 性能提升

| 指标 | 变更前 | 变更后 |
|------|--------|--------|
| WhatsApp 冷启动 | "几十秒" | "几秒" |
| 插件加载方式 | 运行时 TypeScript 编译 | 预编译 JS 直接加载 |

冷启动时间的大幅缩短对生产环境意义重大——当 Gateway 因更新或故障重启时，用户等待 Agent 恢复响应的时间从"可感知的中断"变为"几乎无感"。

## 懒加载路径优化

除了编译插件加载，v2026.3.22 还引入了 **7 条懒加载路径**，将非关键模块的加载推迟到实际需要时：

- **CLI 命令**：频道添加和根帮助路径懒加载，CLI 启动不再加载所有子命令
- **Provider 模块**：Discord 等平台的 provider/session 模块懒加载，未配置的平台不会拖慢启动
- **媒体处理**：图片压缩、音频转码等媒体处理库按需加载

这 7 条懒加载路径与编译插件加载共同作用，形成了"**只加载必要的，延迟加载可选的**"的启动策略。

## 模型目录缓存

v2026.3.22 还对模型目录查询进行了缓存优化。[[Provider-Plugin 架构]] 中每个 Provider Plugin 的 `listModels()` 结果现在按 **配置文件状态和认证文件状态** 进行缓存——只有当配置或认证信息发生变化时才重新查询提供商，否则直接返回缓存结果。

这消除了 Gateway 重启时对所有模型提供商的重复查询，进一步缩短了启动时间。对于配置了多个远程提供商（OpenAI + Anthropic + MiniMax 等）的部署场景，缓存效果尤为显著。

## 与整体架构的关系

编译插件加载是 [[Plugin 扩展系统]] 的性能层面的补充——插件的声明式发现和动态加载机制不变，但加载的产物从源码变为编译产物。这类似于 Java 世界从"每次启动都编译 .java"到"直接加载 .class"的转变。

对于第三方 Plugin 开发者，这也意味着发布到 npm 的插件包应该包含编译后的 `dist/` 目录，而不是仅提供 TypeScript 源码。
