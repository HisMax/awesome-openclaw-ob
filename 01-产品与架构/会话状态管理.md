# 会话状态管理

## 一句话总结

OpenClaw 用 SQLite 持久化每个会话的完整状态（`~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl`），每个会话有严格的生命周期管理，且同一用户在不同平台拥有独立隔离的 Session——会话文件可安全删除，系统会按需重建。

## 会话管理的本质

想象你去一家咖啡店。第一次去，你点了"燕麦拿铁，少冰"。第二天再去，如果店员记得你昨天的单，直接问"老样子？"——这就是**有状态的会话管理**。如果每次都要从头说一遍——这就是**无状态API调用**。

会话管理的本质就是：**让对话拥有记忆和连续性**。它回答一个根本问题：下一轮对话时，系统还认识你吗？

## 为什么 AI Agent 需要会话管理

普通的 LLM API 调用是无状态的——每次请求都是一张白纸。但 AI Agent 不是"问答机"，它是"持续合作的伙伴"。没有会话管理，Agent 就像一个每30秒就失忆的人。

会话管理解决三个核心问题：
1. **上下文连续性**：多轮对话中保持话题一致
2. **任务持久化**：长时间任务中断后能恢复
3. **身份识别**：跨渠道认出"你还是你"

## 核心机制：SQLite 持久化

每个会话存储为 JSONL 文件（`~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl`），每行一条消息记录。JSONL 追加写入性能高，无需加载整个文件就能写入新消息。

### 会话键（Session Key）设计

| 会话类型 | 键格式 | 说明 |
|---------|--------|------|
| DM 会话 | `agent:<agentId>:<mainKey>` | 同一人在 WhatsApp 和 Telegram 聊天，共享同一个会话 |
| 群聊会话 | `agent:<agentId>:<channel>:group:<id>` | 每个群独立隔离 |
| 跨渠道身份 | `session.identityLinks` | `telegram:123` ↔ `whatsapp:456` 映射到同一个人 |

## OpenClaw 的会话管理实现

OpenClaw 的 Gateway 是会话管理的中枢，所有会话通过 WebSocket 连接到 `127.0.0.1:18789`。

**队列串行化**：每个会话内的消息严格串行处理——**每个会话同一时间只能有一个 Agent 运行实例**。如果新消息在 Agent 运行中途到达，系统会 hold-inject-followup（暂存→注入→后续执行），绝不丢失，与[[Steering 与 Follow-Up 控制]]的 followUp 机制直接关联。

## Session 生命周期

```
创建 → 激活 → 对话中(消息按队列串行处理) → 空闲 → 持久化存储 → 恢复/销毁
```

每个阶段：Gateway 创建 Session → 加载到内存 → 按[[Lane-Based Queuing 并发模型|队列]]串行处理消息 → 空闲时卸载到磁盘 → 新消息到达时恢复。**关键特性**：删除会话文件是安全的，系统会按需重建。这种设计让会话既持久又不脆弱。

## 跨平台会话隔离

**同一用户在 WhatsApp 和 Telegram 上拥有两个独立的 Session**，互不影响。

这是一个有意的设计选择：
- **隔离原因**：不同平台的对话上下文可能完全不同（工作 vs 私人）
- **身份关联**：通过 `identityLinks` 可以识别同一用户的不同平台身份
- **记忆共享**：虽然 Session 隔离，但[[记忆系统|持久记忆]]（MEMORY.md、向量数据库）是跨 Session 共享的

```
用户 "张三"
  ├── WhatsApp Session（工作对话）  ← 独立上下文
  ├── Telegram Session（私人对话）  ← 独立上下文
  └── 共享记忆层（MEMORY.md + SQLite 向量库）
```

## 与上下文管理的关系

会话管理负责"**谁在说、说了什么、说到哪了**"，而[[上下文管理机制]]负责"**每次调用 LLM 时塞什么进去**"。会话是容器，上下文是内容。会话管理维护完整历史，上下文管理从中筛选最相关的部分送给模型。

## 后续演进：Session Branching（v2026.4）

v2026.4.7 引入 **Session Branching**——允许在对话中途"分叉"，尝试一个风险操作，然后在分支失败时恢复到之前的状态。这对 [[Agentic Coding]] 场景特别有价值：Agent 可以在隔离的分支中尝试激进的代码修改，失败了回退，成功了合并。详见 [[OpenClaw v2026.4 版本更新]]。

## 关键洞察

会话状态管理的设计揭示了 Agent 系统与传统 Web 应用的本质区别：Web 应用的 Session 管理的是"登录状态"，Agent 的 Session 管理的是"对话记忆"。前者是二值的（登录/未登录），后者是连续的（从第一句话到最后一句话的完整上下文）。JSONL 的追加写入模式、安全删除重建机制、跨平台隔离策略——这些都是为"对话记忆"这个独特需求量身定制的工程决策。
