# 每网关会话隔离

## 一句话理解

> 想象一栋公寓楼里每户共用一个信箱——张三的信件和李四的混在一起，拿错信是迟早的事。每网关会话隔离就是给每户配了独立信箱——每个 Gateway 实例的会话和设置都锁定在自己的"邮编"下，互不干扰。

## 问题背景

在 v2026.3.22 之前，OpenClaw 的 [[Dashboard 控制面板]] 使用全局的 localStorage 键来持久化用户设置（如选中的会话、聊天历史滚动位置、视图模式等）。在**单 Gateway** 部署中这没有问题，但在**多 Gateway** 部署中会出现严重的状态冲突：

- 用户连接 Gateway A 时选中了会话 X，切换到 Gateway B 后，Dashboard 仍然尝试加载会话 X——但这个会话在 Gateway B 上不存在
- 两个 Gateway 的设置互相覆盖，用户在 Gateway A 上的偏好设置会被 Gateway B 的操作刷掉
- 聊天历史范围无法区分，搜索结果混入其他 Gateway 的消息

## 解决方案

v2026.3.22 引入了 **按 Gateway 基路径（base path）作用域化** 的持久化设置键。所有 localStorage 键现在带有 Gateway 的基路径前缀：

```
旧键: openclaw.selectedSession → "session-abc-123"
新键: openclaw./gateway-a/.selectedSession → "session-abc-123"
      openclaw./gateway-b/.selectedSession → "session-xyz-789"
```

### 迁移支持

对于现有的单 Gateway 部署，系统提供了自动迁移——首次连接时，无前缀的旧键会被读取并自动迁移到带前缀的新键格式，然后删除旧键。整个过程对用户透明。

## 功能细节

### 每网关会话选择

[[Dashboard 控制面板]] 的 Session 模块现在维护独立的"当前选中会话"状态。切换 Gateway 时，Dashboard 自动恢复该 Gateway 上次选中的会话，而不是保持前一个 Gateway 的选择。

### 历史范围限定

聊天历史浏览和搜索现在严格限定在当前连接的 Gateway 范围内。这与 [[会话状态管理]] 中的跨平台会话隔离理念一致——不同 Gateway 上的对话上下文可能完全不同，混合展示会造成混淆。

## 适用场景

- **开发/生产分离**：开发者同时连接开发 Gateway 和生产 Gateway，设置互不影响
- **多 Agent 部署**：不同 Agent 运行在不同 Gateway 上，Dashboard 可以在它们之间无缝切换
- **团队共享**：多人通过同一浏览器（不同标签页）访问不同 Gateway
