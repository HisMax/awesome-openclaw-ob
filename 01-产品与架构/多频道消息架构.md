# 多频道消息架构

## 一句话总结

OpenClaw 用一个 Gateway 进程（WebSocket 服务器，`ws://127.0.0.1:18789`）统一接管 22+ 通讯平台的消息，每个平台通过独立的 Channel Adapter 转换为标准内部格式——同一个 Agent 同时在所有平台可达，共享同一份记忆和上下文。

## 核心机制：Hub-and-Spoke 架构

Gateway 是所有组件的中央协调枢纽，采用 Hub-and-Spoke（轮毂-辐条）模式：

```
         WhatsApp ──┐
        (Baileys)   │
                    │
       Telegram ────┤
       (grammY)     │
                    ├──→ Gateway (ws://127.0.0.1:18789)
        Discord ────┤     ├── 消息路由
       (discord.js) │     ├── 认证鉴权
                    │     ├── 会话管理
          Slack ────┤     ├── 工具调度
         (Bolt)     │     └── 状态管理
                    │
        Signal ─────┘
      (signal-cli)
```

### Gateway 职责（Layer 1：会话管理层）

- 管理完整的用户[[会话状态管理|会话]]生命周期
- 消息队列 + 调度优先级（并发控制，最多 10 个并行请求）
- 认证与权限控制
- WebSocket 持久连接（30 秒心跳检测，指数退避重连）

### Channel Adapter 职责（Layer 2：平台适配层）

每个适配器负责：认证（QR 码/Bot Token）→ 入站解析（提取文本、媒体、线程上下文）→ 访问控制（允许名单、DM 配对）→ 出站格式化（平台 Markdown、消息分块、媒体上传）。

## 会话隔离策略

**关键设计**：同一用户在 WhatsApp 和 Telegram 上拥有**两个独立的 Session**，互不影响。

会话键结构：
- DM 会话：`agent:<agentId>:<mainKey>`
- 群聊会话：`agent:<agentId>:<channel>:group:<id>`
- 跨渠道身份：通过 `session.identityLinks` 映射（如 `telegram:123` ↔ `whatsapp:456`）

这意味着用户在 WhatsApp 上说的话，Agent 在 Telegram 上也记得——但它们是不同的会话容器。

## 支持的平台与多 Agent 路由

支持 22+ 平台：WhatsApp（Baileys）、Telegram（grammY）、Discord（discord.js）、Slack（Bolt）、Signal（signal-cli）、iMessage（BlueBubbles）、飞书、Matrix、IRC、Nostr、Teams 等。通过[[Plugin 扩展系统]]可注册新的 Channel Adapter。

一个 Gateway 实例可托管多个 Agent，不同频道/对话路由到隔离的 Agent 实例，各自拥有独立的工作空间、模型配置和行为。

## 后续发展（v2026.5-6）

- **v2026.5.3**：统一流式进度配置——单一 `streaming.mode: "progress"` 值在 Discord、Telegram、Slack、Matrix 和 Teams 上行为完全一致
- **v2026.5.12**：WhatsApp、Slack 等频道从核心安装移出，改为按需安装，新安装更小
- **v2026.5.12**：Telegram 可靠性全面加固——轮询改为独立 worker、持久化本地 spool、支持的 HTML 标签保留
- **v2026.6.2**：Telegram 重复转录镜像修复、飞书审批白名单、Discord 语音错误处理、WhatsApp 出站交付路径安全等广泛可靠性修复

详见 [[OpenClaw v2026.5 版本更新]] 和 [[OpenClaw v2026.6 版本更新]]。

## 关键洞察

多频道架构是 OpenClaw 实现"可及性突破"的技术基础——用户不需要学习新界面，用什么聊天软件就在什么上面用。这解释了为什么 OpenClaw 能在非技术用户中快速传播：**它把 AI Agent 的入口放在了人们每天已经在用的地方**。从工程角度看，每个适配器运行在独立轻量线程中，添加新频道不会降低现有频道性能。
