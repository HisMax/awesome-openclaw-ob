# 上下文引擎 modelId 适配

## 一句话理解

> 给不同型号的"大脑"准备不同尺寸的"工作台"——GPT-5.4 的上下文窗口 256K、Claude Opus 4.6 的 1M，同一个任务需要完全不同的上下文组装策略，v2026.3.22 让上下文引擎能"看到"正在使用哪个模型。

## 核心变更

v2026.3.22 中，`assemble()` 方法新增接收嵌入运行器的 `modelId` 参数：

```
context = engine.assemble(messages, tools, modelId)
```

这让上下文引擎在组装提示词时可以根据目标模型做出不同决策。

## 具体适配行为

### Token 预算管理

- 大窗口模型（如 Claude 1M）：可以保留更多历史对话和工具描述
- 小窗口模型（如本地 Ollama 模型）：激进压缩，优先保留最近交互

### 紧凑目录回退

当 Skills 目录超出提示词预算时，引擎保留所有已注册 Skills 的摘要但裁剪详细描述。这个"回退点"因模型不同而异——大模型可以晚些触发回退。

### 转录维护

转录维护重写保留活跃分支元数据，在压缩（[[Context Compaction]]）后触发溢出恢复。上下文压缩后的转录修复也因模型能力不同而调整策略。

## 为什么重要

在 [[模型无关架构]] 的理念下，OpenClaw 支持几十个不同能力的 LLM。但"模型无关"不等于"一视同仁"——用一个为 4K 上下文优化的策略去组装 1M 上下文模型的提示词，就像用小学课本教大学课程，浪费了能力也错过了深度。
