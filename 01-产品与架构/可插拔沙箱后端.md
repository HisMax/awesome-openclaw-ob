# 可插拔沙箱后端

## 一句话理解

> 可插拔沙箱后端让 OpenClaw 的代码执行环境从"只能在本地跑"变成"想在哪跑就在哪跑"——本地 Docker、远程 SSH 服务器、OpenShell 云端沙箱，通过统一接口随意切换，像选电池一样选执行后端。

## 为什么需要可插拔沙箱

OpenClaw 早期的代码执行直接跑在宿主机上，受 [[自主执行与人机协作|权限等级]] 控制但缺乏物理隔离。v2026.3.22-beta.1 引入可插拔沙箱后端，解决三个核心问题：

1. **安全隔离**：Agent 执行的代码在独立环境中运行，即使恶意代码也无法影响宿主机
2. **环境一致性**：不同用户、不同任务使用标准化的执行环境，消除"在我机器上能跑"问题
3. **部署灵活性**：本地开发用 Docker，团队协作用远程 SSH，云端部署用 OpenShell

## 架构设计

```
Agent Execution Loop
  └── Tool Use（代码执行请求）
        └── Sandbox Router（选择后端）
              ├── Local Backend（本地 Docker/进程隔离）
              ├── OpenShell Backend
              │     ├── mirror 模式（本地文件同步到云端）
              │     └── remote 模式（完全远程工作区）
              └── SSH Backend（远程服务器执行）
```

### OpenShell 后端

OpenShell 是 v2026.3.22-beta.1 引入的核心沙箱后端，支持两种工作模式：

**Mirror 模式**：将本地工作区的文件系统镜像到云端沙箱，Agent 在云端执行代码，结果同步回本地。适合需要强隔离但希望保持本地文件管理的场景。

**Remote 模式**：完全在远程工作区中操作，无需本地文件。适合云端部署、CI/CD 流水线和团队共享环境。

### SSH 后端

核心 SSH 后端用于远程执行，支持标准 SSH 密钥认证。适合已有服务器资源的团队，无需额外的沙箱服务即可获得远程隔离执行能力。

## 与现有架构的关系

可插拔沙箱后端遵循 OpenClaw 一贯的 [[模型无关架构|"抽象层 + 适配器"]] 设计模式——正如 Provider 插件让模型可替换，沙箱后端让执行环境可替换。两者通过 [[Plugin 扩展系统]] 的统一加载机制管理。

在 [[Agent Execution Loop]] 中，工具调用的执行阶段现在会经过 Sandbox Router，根据配置选择合适的后端。这不改变 Agent 的逻辑流程，只改变"代码在哪跑"。

## 安全考量

- OpenShell 后端提供了比本地 Docker 更强的隔离——云端沙箱与用户主机完全网络隔离
- SSH 后端的安全性取决于远程服务器的配置，建议使用专用的低权限用户
- Mirror 模式存在文件同步的安全敏感性——需注意 `.env` 等敏感文件的排除规则
- 可插拔设计意味着未来可以接入更多沙箱后端（如 Firecracker microVM、gVisor 等）

## 关键洞察

可插拔沙箱后端是 OpenClaw 从"个人助手"向"企业级 Agent 平台"演进的关键一步。个人用户可以继续用本地执行获得最快速度，企业可以用 OpenShell 获得合规隔离，DevOps 团队可以用 SSH 后端对接现有基础设施。这种"同一套 Agent 逻辑，不同执行环境"的设计，让 OpenClaw 的部署边界从"一台电脑"扩展到"任何有计算能力的地方"。
