# jq 安全白名单移除

## 一句话理解

> jq 看起来只是一把裁纸刀——用来裁剪 JSON 数据，人畜无害。但 `jq -n env` 这条命令能把系统所有环境变量倒出来，就像那把裁纸刀其实能打开保险箱。v2026.3.22 把这把刀从"随便用"的工具架上撤走了。

## 发现过程：一条命令泄露所有秘密

`jq` 是 JSON 处理的瑞士军刀，几乎是每个开发者的标配工具。在 OpenClaw 的执行白名单中，`jq` 一直被归类为"安全二进制"（safe-bin），Agent 可以无需人工审批直接调用。

然而 `jq` 内置了一个鲜为人知的功能：

```bash
jq -n env
```

这条命令会将当前进程的**全部环境变量**以 JSON 格式输出。环境变量中通常包含：

| 泄露内容 | 典型变量名 | 危害等级 |
|----------|-----------|---------|
| API 密钥 | `OPENAI_API_KEY`、`ANTHROPIC_API_KEY` | 严重——直接盗用 API 配额 |
| 数据库凭证 | `DATABASE_URL`、`REDIS_URL` | 严重——数据库完全暴露 |
| 云平台凭证 | `AWS_SECRET_ACCESS_KEY`、`AZURE_CLIENT_SECRET` | 严重——云账户接管 |
| 内部令牌 | `GITHUB_TOKEN`、`SLACK_BOT_TOKEN` | 高——横向移动 |
| 路径信息 | `PATH`、`HOME`、`USER` | 中——信息侦查 |

这意味着一个被 [[Prompt Injection 风险|提示注入]] 操控的 Agent 只需执行一条白名单内的命令，就能获取系统中所有通过环境变量传递的敏感信息——无需任何越权操作，因为 `jq` 本就在白名单中。

## 白名单移除与附加加固

v2026.3.22 做了两项修复：

### jq 移出 safe-bin 白名单

`jq` 从默认安全二进制白名单中移除。Agent 调用 `jq` 时将触发人工审批流程。用户可以在审批提示中看到完整的 `jq` 命令及参数，决定是否允许执行。

### exec 白名单的模式检查加固

在审查 `jq` 问题的过程中，开发团队发现 exec 白名单还存在其他可利用的模式：

- **单引号 `$0`/`$n` 令牌**：攻击者可以通过 `$0` 引用当前 shell 的路径，在白名单检查中构造误导性的命令字符串
- **换行符分隔的 exec**：将多条命令用换行符拼接在一条白名单命令中，绕过逐行检查

这两类模式现在也被 exec 白名单拒绝，形成了 [[macOS 执行白名单加固]] 中描述的白名单防护体系的一部分。

## 更广泛的教训：无害工具的武器化

`jq` 的案例是 [[凭证泄露与信息窃取|信息泄露]] 攻击的经典范式——攻击不依赖漏洞，而是利用合法工具的合法功能。类似的例子在安全领域屡见不鲜：

- `curl` 可以将文件内容 POST 到远程服务器
- `python -c` 可以执行任意代码
- `git` 的 `core.fsmonitor` 配置可以执行任意命令

对于 AI Agent 来说，这个问题尤为严峻：Agent 不理解"调用 `jq -n env` 会泄露敏感信息"这一安全含义，它只知道 `jq` 在白名单中可以直接使用。这再次印证了 [[致命三合一安全矛盾]]——赋予 Agent 越多的工具使用权限，信息泄露的风险就越大。
