# SSRF 固定与代理安全

## 一句话理解

> SSRF 就像让一个可信的信使帮你送信，但信使不检查信封里装的是什么——攻击者把"去内部服务器拿机密"写进信封，信使照办不误。SSRF 固定就是给信使装上X光机，在送出去之前扫描每一封信的真实目的地。

## 什么是 SSRF 以及为什么 OpenClaw 特别脆弱

服务端请求伪造（SSRF）允许攻击者利用服务器作为跳板，访问原本不可达的内部资源。[[OpenClaw 是什么|OpenClaw]] 的 Agent 天然需要发起网络请求（调用 API、下载资源、访问 Webhook），这使得 SSRF 攻击面远大于传统 Web 应用。

在 [[2026年3月安全公告汇总]] 的"其他安全改进"中，网络 SSRF 风险修复被列为重点项。v2026.3.22 则进一步引入了系统性的 SSRF 防护体系。

## 显式代理 SSRF 固定

当 OpenClaw 配置了显式代理（explicit proxy）时，传统的 DNS rebinding 防护会失效——因为请求先到代理，由代理负责 DNS 解析，服务端无法在连接前验证真实目标 IP。v2026.3.22 的解决方案是 **SSRF pinning with HTTPS proxy tunnel translation**：

1. **连接前验证**：在发起代理请求前，先对目标主机进行独立 DNS 解析并校验 IP 地址是否落在内部网段（`127.0.0.0/8`、`10.0.0.0/8`、`169.254.0.0/16`、`fc00::/7` 等）
2. **HTTPS 隧道翻译**：将 CONNECT 隧道请求的目标与预解析的 IP 进行绑定，防止代理在隧道建立后被 DNS rebinding 指向内网地址
3. **响应校验**：代理返回的响应头中的 Location 重定向目标同样需通过 SSRF 固定检查

## 本地文件系统防护

SSRF 不仅限于网络请求。v2026.3.22 封堵了两类本地资源访问：

- **`file://` 协议阻断**：远程主机提供的媒体 URL 如果使用 `file://` 协议，在触达本地文件系统解析之前即被拒绝。这防止了攻击者通过消息平台发送 `file:///etc/passwd` 之类的链接来读取服务器文件
- **UNC/网络路径拦截**：Windows 环境下的 `\\server\share` 路径在解析前被阻断，直接切断了 SMB 凭证握手的可能性——不让系统发起 NTLM 认证请求，就不会有 NTLMv2 哈希被中继或破解

这与 [[安全边界与风险（总览）]] 中描述的"攻击面扩大"趋势一致：随着 OpenClaw 接入越来越多的消息平台（Telegram、iMessage、飞书等），每个平台的媒体处理都可能成为 SSRF 入口。

## Windows SMB 凭证握手防护

Windows 环境下，访问 UNC 路径（如 `\\attacker.com\share`）会触发系统自动发起 SMB 认证握手，将当前用户的 NTLMv2 凭证哈希发送给远程服务器。v2026.3.22 在路径解析层面阻止了这一行为，确保即使 Agent 收到包含 UNC 路径的指令，也不会泄露 Windows 凭证。

## Bonjour/DNS-SD 故障处理

在 macOS 环境中，Bonjour（DNS-SD）服务发现失败时可能产生意外的 DNS 查询。v2026.3.22 对 Bonjour 解析失败进行了优雅处理，避免因服务发现超时而产生可被利用的 DNS 侧信道。
