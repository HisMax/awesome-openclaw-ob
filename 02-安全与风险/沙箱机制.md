# 沙箱机制


![[assets/sandbox.jpg]]

[[OpenClaw 是什么|OpenClaw]] 使用 Docker 容器作为沙箱隔离机制，但存在重大限制。

## 现状

- 沙箱使用 Docker 容器实现
- **默认关闭**，需要用户手动开启——这是 [[致命三合一安全矛盾]] 的直接体现
- 早期版本默认绑定到 `0.0.0.0:18789`，直接暴露在网络上（参见大规模实例暴露）

## 已知绕过方式

1. **工具策略绕过**：利用工具注册和执行之间的逻辑漏洞
2. **TOCTOU 竞态条件**：约 **25% 暴力成功率**绕过安全检查
3. **`tools.exec.host` = `gateway`**：通过 API 设置强制命令在宿主机执行，直接逃出容器——[[权限控制机制]]的关键缺陷
4. **[[ClawJacked 远程代码执行漏洞]]**：权限提升后可完全绕过沙箱
5. **[[Claw Chain 四漏洞链]]**（2026 年 5 月）：OpenShell 文件系统桥接层的 TOCTOU 读/写逃逸（CVE-2026-44112/44113），实现从沙箱内到宿主机的完整 kill chain，影响 **245,000** 个暴露实例

> Snyk Labs 已发布完整的沙箱绕过研究报告。

## 与 [[Prompt Injection 风险]] 的关系

沙箱绕过与提示注入攻击结合时尤为危险——攻击者可以通过恶意 Skill 诱导 Agent 执行逃逸容器的命令。数据泄露在沙箱被绕过后会急剧上升，因为 Agent 可以访问宿主机上的所有凭证和敏感数据。

## 替代方案

物理隔离策略——每个 Agent 运行在独立的 Mac Mini 上，从根本上避免了容器逃逸问题。

## 各方建议

| 来源 | 建议 |
|------|------|
| **Sophos** | 仅在**一次性沙箱（disposable sandbox）**中运行 |
| **Microsoft** | 部署在**完全隔离环境** |
| **Kaspersky** | 使用**专用备用计算机或 VPS** |
| **Aikido.dev** | 全面强化基本上把它变成了带额外编排的 ChatGPT |
