# macOS 执行白名单加固

## 一句话理解

> 执行白名单就像机场安检名单——只有名单上的人能登机。但如果有人穿上机长制服（wrapper 欺骗）、用假名字办了登机牌（env 路径劫持）、甚至在名字里混入肉眼看不见的隐形字符（韩文填充码点），安检就形同虚设。v2026.3.22 把这些伪装手段一一识破。

## 白名单机制与 OpenClaw 的自主执行

[[自主执行与人机协作|OpenClaw 的自主执行]] 机制允许 Agent 在不同权限等级下运行命令。在需要人类审批的等级中，**执行白名单**（allowlist）决定了哪些命令可以自动放行、哪些需要弹出审批提示。macOS 上的白名单解析尤其复杂，因为 macOS 的 PATH 环境、命令别名、wrapper 脚本层层嵌套，给攻击者留下了大量可利用的缝隙。

## v2026.3.22 封堵的三类欺骗

### 1. Wrapper 与 env 欺骗

攻击者可以通过以下方式绕过白名单检查：

- **Wrapper 脚本**：将恶意命令包装在白名单命令的 wrapper 中。例如 `/usr/bin/time` 是一个常见的透明 wrapper，如果白名单只检查最外层命令名，`time malicious_command` 就能通过
- **`env` 命令劫持**：`env -i PATH=/attacker/bin command` 可以替换整个 PATH 环境，使同名命令指向恶意二进制
- **符号链接伪装**：创建 symlink 让恶意二进制看起来像白名单中的合法命令

v2026.3.22 的修复：
- `time` 在白名单评估中被视为**透明包装器**——系统会穿透 `time` 直接检查其后的实际命令
- `env` 欺骗的 PATH 覆盖被检测并阻断
- 白名单解析不再仅依赖命令名称，而是解析到最终的二进制路径

### 2. 韩文空白填充码点欺骗

Unicode 中存在"空白填充码点"（Blank Hangul Filler，U+3164 等），这些字符在终端和审批提示中**完全不可见**，但在字符串比较中是不同的字符。攻击者可以在命令名中插入这些码点，让审批提示显示的命令看起来与白名单中的完全一致，但实际上指向不同的二进制。

v2026.3.22 在审批提示渲染时对韩文填充码点进行**转义显示**，确保用户能看到命令中隐藏的不可见字符，从而做出正确的审批决策。这与 [[2026年3月安全公告汇总]] 中提到的"Unicode 混淆攻击防护"属于同一防护体系。

### 3. 宿主环境变量覆盖

在 OpenClaw 的 Gateway-Node 架构中，环境变量可能从宿主机传递到执行节点。攻击者如果能控制宿主机的环境变量，就可能覆盖 Node 上的 PATH 或 LD_PRELOAD，间接影响白名单解析结果。

v2026.3.22 统一了 Gateway 和 Node 之间的环境变量处理逻辑，确保白名单解析使用的是经过验证的环境，而非直接继承宿主环境。这一加固对 [[可插拔沙箱后端]] 的 SSH 后端尤为重要——SSH 连接天然会传递客户端的环境变量。

## 为什么 macOS 是重灾区

macOS 是 OpenClaw 用户密度最高的平台。与 Linux 不同，macOS 的 `/usr/bin/` 下大量命令是 wrapper 或 shim（如 Xcode Command Line Tools 的 wrapper），PATH 中 Homebrew、MacPorts、nix 等包管理器的路径交织，加上系统完整性保护（SIP）限制了某些安全加固手段——这使得 macOS 上的白名单解析比 Linux 复杂一个数量级。
