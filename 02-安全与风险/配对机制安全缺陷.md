# 配对机制安全缺陷

**一句话总结**：OpenClaw 的设备配对机制存在两个相互关联的安全缺陷——引导配对码可被重放提权至 `operator.admin`，且 `/pair` 命令将永久凭证嵌入配对载荷。

本笔记涵盖两个 GHSA 公告：
- **GHSA-63f5-hhc7-cx6p**（高危）：引导配对码重放提权
- **GHSA-7h7g-x2px-94hj**（中危）：配对设置码泄露长期凭证

## 漏洞一：引导配对码重放提权

### 基本信息

| 项目 | 详情 |
|------|------|
| GHSA 编号 | GHSA-63f5-hhc7-cx6p |
| 严重度 | **高危（High）** |
| 影响版本 | <=2026.3.12 |
| 修复版本 | 2026.3.13 |
| 漏洞类型 | 认证绕过 / 权限提升 |

### 漏洞分析

OpenClaw 的引导（bootstrap）流程使用一次性令牌完成初始设备配对。然而，该令牌在使用后**未被正确失效**，攻击者截获 bootstrap 令牌后可反复重放，每次重放都能将权限提升至 `operator.admin`。

攻击路径：
1. 攻击者通过网络嗅探、日志泄露或社会工程获取 bootstrap 令牌
2. 使用该令牌发起配对请求
3. 获得 `operator.admin` 权限
4. 令牌未失效，可重复使用——即使管理员撤销了攻击者的设备，攻击者仍可重新配对

### 与 CVE-2026-25253 修复的矛盾

讽刺的是，TOFU（Trust On First Use）策略是 [[CVE-2026-25253 ClawJacked 漏洞详解|ClawJacked 漏洞]] 修复后引入的安全机制。但配对码重放缺陷表明，TOFU 策略的实现本身存在问题——初始信任建立过程中的令牌生命周期管理不完善。

## 漏洞二：配对设置码泄露长期凭证

### 基本信息

| 项目 | 详情 |
|------|------|
| GHSA 编号 | GHSA-7h7g-x2px-94hj |
| 严重度 | **中危（Moderate）** |
| 影响版本 | <=2026.3.11 |
| 修复版本 | 2026.3.12 |
| 漏洞类型 | 信息泄露 |

### 漏洞分析

OpenClaw 的 `/pair` 命令在生成配对载荷时，将**永久凭证**嵌入其中。这意味着：

- 配对码本应是短期、一次性的
- 但载荷中包含的凭证是**长期有效**的
- 任何获取到配对载荷的人都能提取出永久凭证
- 即使配对流程完成后，这些凭证仍然有效

### 与凭证泄露问题的关系

此漏洞与 [[凭证泄露与信息窃取]] 中描述的明文凭证存储问题属于同一类——OpenClaw 在多处将敏感凭证暴露在不必要的位置。配对载荷中嵌入永久凭证，相当于将密钥印在了临时通行证上。

## 两个漏洞的组合效应

两个配对机制缺陷组合使用时，危害显著放大：

```
攻击者获取配对载荷
     │
     ├──→ 提取永久凭证（GHSA-7h7g）
     │         │
     │         └──→ 长期持久化访问
     │
     └──→ 重放 bootstrap 令牌（GHSA-63f5）
               │
               └──→ 反复提权至 operator.admin
                         │
                         └──→ 完全控制网关
```

## 修复措施

### GHSA-63f5 修复（2026.3.13）
- Bootstrap 令牌使用后**立即失效**
- 引入令牌使用计数器，强制一次性使用
- 令牌绑定时间窗口，过期自动失效

### GHSA-7h7g 修复（2026.3.12）
- `/pair` 命令生成的载荷中**不再包含永久凭证**
- 配对载荷仅包含短期、用途受限的临时令牌
- 永久凭证仅在配对完成后通过安全通道下发

## 安全建议

1. **立即升级**至 2026.3.13 或更高版本
2. **轮换所有配对相关凭证**
3. 审计已配对设备列表，移除未知设备
4. 检查日志中是否有异常的重复配对请求
5. 避免通过不安全渠道传输配对码

## 后续发展：CVE-2026-32922

配对机制的安全问题在 CVE-2026-32922（CVSS 9.9）中再次暴露——`rotateDeviceToken` 函数未校验请求的 scope 是否为调用者已有 scope 的子集，单个 `operator.pairing` token 即可提升为 `operator.admin` + RCE。被 ARMO 称为"OpenClaw 历史上最严重的漏洞"。已在 v2026.3.11 中通过原子 token 轮换修复。
