# 沙箱运行时注入防护

## 一句话理解

> 沙箱本应是一间密封的玻璃房，但构建工具的环境变量就像墙上没封死的通风管道——攻击者往 `MAVEN_OPTS` 里塞一段 `-javaagent`，JVM 就乖乖把恶意代码带出了房间。v2026.3.22 把这些管道全部焊死了。

## 背景：为什么环境变量是逃逸通道

[[可插拔沙箱后端]]为 OpenClaw 提供了 Docker、SSH、OpenShell 等多种隔离执行环境。然而沙箱的安全边界取决于**最薄弱的泄露路径**。JVM 系构建工具（Maven、SBT、Gradle、Ant）都会在启动时读取 `*_OPTS` 环境变量并将其作为 JVM 参数注入——这意味着攻击者只需在沙箱内设置一个环境变量，就能通过 `-javaagent`、`-XX:OnOutOfMemoryError` 等 JVM 开关执行任意代码，甚至加载宿主机上的 native library，从而绕过沙箱的进程隔离。

同样的逻辑适用于：

| 注入向量 | 机制 | 风险 |
|----------|------|------|
| `MAVEN_OPTS` / `SBT_OPTS` / `GRADLE_OPTS` / `ANT_OPTS` | JVM 启动参数注入 | 任意代码执行、类加载劫持 |
| `GLIBC_TUNABLES` | glibc 动态链接器可调参数 | CVE-2023-4911 风格的本地提权 |
| `DOTNET_ADDITIONAL_DEPS` | .NET 运行时额外依赖注入 | DLL 劫持、依赖链污染 |

这些都不是理论漏洞——它们是已知的、在 CTF 和实际渗透中反复利用的技术。在 [[2026年3月安全公告汇总]] 记录的 v2026.3.22 安全加固中，这些注入向量被明确列为封堵对象。

## v2026.3.22 的封堵策略

[[OpenClaw v2026.3 版本更新|v2026.3.22]] 在沙箱初始化阶段引入了环境变量黑名单机制：

1. **JVM 构建工具系列**：`MAVEN_OPTS`、`SBT_OPTS`、`GRADLE_OPTS`、`ANT_OPTS` 在沙箱内被强制清除或设为只读空值，无论用户或 Skill 如何设置都不会传递给子进程
2. **Glibc 可调参数**：`GLIBC_TUNABLES` 被完全阻断，防止通过动态链接器的调参接口触发堆溢出或越界写入
3. **.NET 依赖劫持**：`DOTNET_ADDITIONAL_DEPS` 被过滤，防止攻击者在 .NET 项目构建时注入恶意程序集

这种"黑名单 + 沙箱边界拦截"的双层防御，确保即使 Agent 被 [[Prompt Injection 风险|提示注入]] 诱导执行 `export MAVEN_OPTS=...`，环境变量也无法逃逸到实际的构建进程中。

## 与沙箱整体安全的关系

运行时注入防护是 [[可插拔沙箱后端]] 安全模型的**补完措施**。沙箱后端解决的是"代码在哪跑"（隔离边界），运行时注入防护解决的是"沙箱内的进程能不能偷渡出去"（边界完整性）。两者缺一不可——没有沙箱，注入防护无从谈起；有沙箱但不封堵注入通道，隔离形同虚设。

这也是 [[致命三合一安全矛盾]] 在工程层面的缩影：开发者需要 Maven 和 Gradle 来构建项目，但构建工具的环境变量机制天然就是安全漏洞。v2026.3.22 选择了"在沙箱边界拦截"而非"禁止使用构建工具"，是在安全性与实用性之间的务实平衡。
