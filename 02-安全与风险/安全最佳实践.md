# 安全最佳实践

基于 [[安全边界与风险（总览）]] 中的各类风险，综合各安全厂商建议的防护措施。这些实践适用于所有部署 [[OpenClaw 是什么|OpenClaw]] 的场景。

## 环境隔离

1. **不要在主要工作设备上运行** OpenClaw（Kaspersky）
2. 使用**专用备用计算机或 VPS**
3. 如必须评估，部署在**完全隔离环境**（Microsoft）
4. 仅在**一次性沙箱（disposable sandbox）**中运行（Sophos）

## 沙箱配置

- **必须手动开启** Docker [[沙箱机制|沙箱]]（默认关闭）
- 不要将实例绑定到 `0.0.0.0`，保持 localhost 绑定
- 确保使用加密的 HTTPS 连接

## 权限最小化

- 保持用户确认机制（`exec.approvals.set`）开启
- 不要将 `tools.exec.host` 设为 `gateway`
- 限制 Agent 访问的数据范围
- 遵循 [[权限控制机制|最小权限原则]]

## 凭证安全

- 定期轮换 API 密钥和网关令牌
- 监控 `~/.openclaw/` 目录的异常访问
- 不在配置文件中存储高敏感凭证
- 防范 [[凭证泄露与信息窃取|Infostealer]] 攻击

## Skills 安全

- 审查所有安装的 Skills 的来源和代码
- 避免安装未经验证的第三方 Skills
- 关注 Snyk ToxicSkills 报告中的已知恶意 Skills 列表

## 网络安全

- 确保实例不暴露在公网（防止大规模实例暴露）
- 启用认证机制
- 使用防火墙限制访问

## 版本管理

- 及时更新到最新版本——v2026.4.22 修复了 [[Claw Chain 四漏洞链]]，v2026.6.2 引入 [[Operator Install Policy]]
- 关注安全公告和 CVE 披露——CVE 累计已突破 **472 个**（截至 2026 年 5 月）
- 参考新加坡 CSA（AD-2026-005）等国家级安全指导

## 根本性限制

需要认识到 Prompt Injection 在当前 LLM 架构下**无法根本解决**。[[致命三合一安全矛盾]] 意味着功能越强大的 Agent，安全风险越高。
